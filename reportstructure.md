# CHAPTER 4: SYSTEM DEVELOPMENT (FYP2) 

## 1. Introduction to the chapter
This chapter serves as an exhaustive, highly technical exposition of the system development lifecycle and sophisticated architectural engineering implemented to realize the ShieldGuard Vishing Detection System. Operating at the cutting-edge intersection of applied cybersecurity and natural language processing, ShieldGuard was designed to address the critical, systemic shortcomings of traditional phishing detectors. The project evolved from a rudimentary, monolithic machine learning prototype into a state-of-the-art, fully decoupled, horizontally scalable web application. The core algorithmic architecture hinges entirely upon a complex, sequential system known as the Hybrid Intelligence Cascade. This cascade rigorously pipes unstructured user data—such as live audio recordings or pasted transcript strings—through an intense gauntlet comprised of classical Machine Learning anomaly detection, a Retrieval-Augmented Generation (RAG) vector database, and finally into a highly collaborative Multi-Agent Large Language Model ecosystem engineered via the CrewAI framework. 

## 2. Preparation (Tech Stack, Architecture, & Implementation)

Constructing a fault-tolerant, high-speed detection engine while simultaneously preserving maximum user data privacy mandated that the architectural technology stack be stringently mapped into a secure, decentralized tri-layer structure spanning the Client Interface, the Middleware Logic, and the Persistent AI Storage. 

### 2.1 Explicit Technology Stack & Versions 
*   **Backend / Middleware Layer:**
    *   **Python 3.12+**: Selected for its advanced typing structures and compatibility with modern AI data science ecosystems.
    *   **FastAPI (v0.115+)**: Chosen over Django or Flask due to its native implementation of asynchronous operations and automatic Pydantic JSON validation schemas.
    *   **Uvicorn (v0.32+)**: An ASGI web server bridging the HTTP network requests securely to the FastAPI application.
    *   **Scikit-Learn (v1.7.2)**: Specifically used to deploy the Term Frequency-Inverse Document Frequency (TF-IDF) mathematical vectorizer and the foundational Support Vector Machine (SVM) classification algorithms.
    *   **TensorFlow/Keras (v2.15.1)**: Utilized during the training phases for deep learning Neural Network comparative evaluations.
    *   **OpenAI Whisper (Base Model)**: The primary algorithmic layer for Speech-to-Text inference computation, operating locally without sending audio to OpenAI servers.
    *   **ffmpeg & pydub**: Core binary libraries integrated to handle the physical chunking and parsing of `.webm` and `.mp3` microphone data.
*   **Generative AI & RAG Persistence Layer:**
    *   **Ollama (Binding to Port 11434)**: The local LLM inference engine strictly shielding user data by bypassing all external cloud APIs. 
    *   **Llama 3.2 (3B Parameters)**: The highly quantized Meta neural network chosen for generating contextual semantic explanations.
    *   **ChromaDB (v0.5.0+)**: An explicit, integrated vector database handling the persistence layer for historical vishing scenarios.
    *   **sentence-transformers (`all-MiniLM-L6-v2`)**: The HuggingFace dense embedding model actively deployed to map text strings into 384-dimensional mathematical arrays.
    *   **CrewAI (v0.28.0+) & Langchain**: Orchestration libraries managing the prompt engineering and sequential routing between the four autonomous LLM Agents.
*   **Frontend Client Layer:**
    *   **Node.js (v18+) & Vite (v8.x)**: The foundational build tool executing rapid hot-module replacements and highly compressed production builds.
    *   **React (v18)**: The user interface library utilizing a Virtual DOM to flawlessly render the system state without page reloads.
    *   **Tailwind CSS (v4.x)**: Atomic utility-based classes deployed heavily to construct a responsive, highly specialized cyberpunk-adjacent visual styling layout.
    *   **Zustand**: A global state management hook explicitly deployed to avoid the notoriously heavy boilerplate of Redux when tracking the audio-recording statuses.
    *   **Recharts**: The charting dependency tasked with dynamically rendering SVG data graphics (e.g., the RiskGauge).
    *   **Axios**: Secure HTTP payload management featuring pre-configured interceptors designed to append authorization headers automatically.
*   **Database & Authentication Layer:**
    *   **Supabase (Cloud PostgreSQL)**: A relational persistence database.
    *   **bcrypt**: High-strength password hashing algorithms explicitly enforcing 12 salt rounds per registry payload.
    *   **python-jose**: The cryptographic handler constructing and validating the JSON Web Tokens (JWT) using HS256 signers.

### 2.2 System Architecture Topology
The implementation relies heavily on decoupling core services to prevent single points of failure. The topology forces all incoming traffic through explicit bottlenecks.

***Implementation Flow:***
1.  User interfaces strictly with the Vite/React UI hosted entirely on Port 5173.
2.  Axios HTTP network packets (typically carrying raw audio binaries or transcript text) securely traverse to Port 8000 via a protected `/api/analyze` REST route.
3.  FastAPI middleware sanitizes the text, instantly querying the Scikit-learn `.pkl` ML model mapped dynamically in RAM.
4.  If the text is deemed benign, FastAPI returns the payload instantly to React. If flagged, the backend requests a dense context array calculation from the ChromaDB mapping directory.
5.  With Historical Context in hand, FastAPI sequentially commands local Ollama to boot the 4 distinct CrewAI personas.
6.  The resultant generative synthesis is formatted into JSON dictionaries, and securely relayed back via HTTP response.

## 3. Development – version 1 (Machine Learning Foundation)
The initial development phase commenced by constructing the foundational deterministic mechanism—a Phase 1 Machine Learning classifier capable of rejecting false positives instantly prior to requesting extreme, latency-inducing LLM compute power. 

### 3.1 Implementation of the Disparate Classification Matrix
*   **Data Normalization and Sanitization:** Raw data representing 1,785 uniquely labeled audio scripts were systematically fed directly through python RegEx pipelines. String manipulations were explicitly enforced to lowercase inputs, strip all punctuation (`re.sub(r'[^\w\s]',...)`), and fundamentally drop English linguistic stop-words (e.g., "the", "and") to surgically eliminate computational noise.
*   **TF-IDF Feature Mapping:** Once sanitized, the data was successfully piped into a Term Frequency-Inverse Document Frequency Vectorizer. This implementation mathematically transforms textual word occurrences into massive, static multi-dimensional frequency matrices, granting heavy internal weights to critical phishing keywords (e.g., "OTP", "urgent", "frozen").
*   **Algorithm Validation Arrays:** Python code was drafted to simultaneously train classification boundaries across multiple models. Logistic Regression mapped baseline accuracy; Random Forest trees combated decision biases; and deep-learning Neural Networks handled non-linear variance. 
*   **Conclusion of Phase 1:** Following extensive cross-validation routines mapped across the 80/20 train/test data splits, the `CalibratedClassifierCV` wrapper enveloping the linear Support Vector Machine (SVM) algorithm definitively dominated evaluation matrices, registering a reliable 98.5% validation accuracy quotient. By serializing this model utilizing `joblib` into a lightweight, offline `svm_model.pkl` weight file, the application effectively finalized the algorithmic "gatekeeper," scoring text inputs across the entirety of the database in under 100 milliseconds outright.

## 4. Development – version 2 (Multi-Agent Artificial Intelligence Deployment)
Phase 2 fundamentally transformed the project from a localized data-science scripting routine into a deeply sophisticated, multi-agent artificial intelligence ecosystem providing nuanced explainability. Phase 2 engineered the specific bridging mathematics: if the Phase 1 SVM output generates a numerical threat confidence score `ml_score >= 0.45`, the execution path transitions deeply into the RAG module.

### 4.1 RAG and CrewAI Multi-Agent Implementation Protocols
*   **ChromaDB Vectorization Deployment:** 1,266 vividly detailed historical vishing transcripts were tokenized and embedded mathematically utilizing `all-MiniLM-L6-v2` bindings directly into the localized `scam_library` ChromaDB index. When invoked, a cosine similarity equation executes comparing the new user threat array linearly against the indexed arrays, dynamically returning the Top-2 closest historical analogs to behave as verified template context.
*   **Node-Based CrewAI Agent Orchestration:** To halt fatal LLM hallucination errors typical in static standard prompting, the `agents/crew.py` framework was engineered. It explicitly commands the local `llama3.2:3b` Langchain model to assume four separate, strictly specialized processing identities sequentially:
    *   *The Technical Auditor Node:* Tasked strictly with verifying the SVM output numbers generated in Phase 1 against the semantic context, issuing validation strings protecting against false positives.
    *   *The Pattern Detective Node:* Programmed to systematically ingest the ChromaDB analog arrays to rigidly classify the current transcript into a categorized taxonomy (e.g., "Bank Impersonation", "Tech Support Refund Scam").
    *   *The Psychology Profiler Node:* Designed completely around extracting heavily manipulated intent. It hunts the user tracking string for 14 uniquely programmed Regex anchor groupings representing coercion methodologies (e.g., explicitly labeling URGENCY algorithms when detecting strings like "act now" or AUTHORITY matrices when detecting "tax evasion").
    *   *The Safety Guardian Node:* Consolidates the prior isolated outputs and enforces a highly specific JSON dictionary structured output. The final implementation mandates a specific `VERDICT`, an easy-to-read `EXPLANATION`, and explicitly outlined `ACTION_STEPS`.
*   **The Cross-Check Divergence Implementation:** An autonomous security protocol script was engineered globally across the cascade layer to trap mathematical overrides. If the fast SVM algorithm statistically predicts "VISHING," but the holistic LLM outputs a "SAFE" verification—and the arithmetic difference explicitly crosses the programmed `0.30` threshold—the FastAPI server logic inherently overrides the LLM response container, forcibly rendering a "Suspicious — Unconfirmed" categorization to absolutely protect clients from potential Neural Network malfunctions.
*   **Frontend UI Component Implementation:** React logic directly maps to these findings. The component `RiskGauge.jsx` uses Recharts to visually swing an SVG dial dial based on the SVM severity score. Concurrently, the `XAIPanel.jsx` component dynamically unwraps the JSON dictionary received from the Safety Guardian agent, populating specific arrays (like rendering the "Psychology Profiler" traits brightly inside red CSS `.tacticChips`).

## 5. Summary
In summary, the system development timeline required shifting radically from experimental, monolithic Jupyter and Streamlit notebook architectures into a highly robust, production-ready horizontal server paradigm. Everything from implementing cryptographic PyDantic route validators to fine-tuning the specific memory allocation of massive LLM arrays was entirely required to execute this goal. The project successfully realized a unified, real-time, and completely privacy-respecting cyber defense application application capable of enterprise-level functionality.

# CHAPTER 6: CONCLUSIONS (FYP2) 

## 1. General conclusion of the overall project
The successful culmination of the ShieldGuard engineering project concretely validates the central hypothesis proposed by this thesis: meticulously merging the high-velocity mathematical determinism inherent to traditional Machine Learning algorithms with the highly sophisticated, contextual semantic reasoning properties of Generative Large Language Models creates a substantially superior methodology for detecting Voice Phishing threats. By explicitly rejecting the standard, opaque monolithic programmatic paradigm and strategically engineering a fully decoupled, Tri-Layer web framework, the system expertly mitigates the massive performance limitations negatively impacting generic cybersecurity algorithms today. Furthermore, by enforcing absolute data privacy safely upon local Offline hardware environments through Ollama hosting, the system effectively shields victims from secondary data extrusion vulnerabilities.

## 2. Discuss why the findings is like what you have found
Detailed observational outputs captured from the comprehensive testing suite overwhelmingly prove that stacking divergent algorithm classes in a cascading pipeline fundamentally resolves intrinsically fatal flaws occurring in standalone deployments. When isolated individually, classical Support Vector Machine algorithms map text features incredibly fast but fundamentally struggle to explain why a call sequence is malicious, as they structurally lack the true semantic comprehension required for human readability. Conversely, utilizing LLM frameworks independently generates untenable latency compute bottlenecks and introduces unacceptable probabilities of generative hallucination anomalies. The empirical results explicitly show that positioning the SVM logic aggressively as an unyielding high-speed gatekeeper filter—while simultaneously grounding the subsequent LLM Agents with ChromaDB historical RAG templates—forces the entire artificial intelligence array to contrast raw user threats against definitively verified, historical attack topologies. This novel symbiosis guarantees extreme API throughput speed alongside absolute accuracy depth, preventing the system from guessing, freezing, or stalling. Furthermore, the penetration testing findings similarly demonstrated unyielding security resilience simply due to the intentional decoupling of the React client-side Virtual DOM from the analytical Python routing pools containing the database connection strings; this drastically reduced the available attack surface and structurally neutralized lethal XSS injection vulnerabilities dynamically before execution.

## 3. Describe how you have achieved all of your objectives
The primary operational system objectives established at the onset of this project were fulfilled completely through dedicated programmatic milestones implemented sequentially across the timeline. Objective 1—to attain ultra-high classification accuracy inside strict real-time metrics—was achieved directly via the deployment of the initial Layer 1 ML module. This specific module isolates heavy text workloads directly onto a pre-initialized, memory-loaded TF-IDF SVM configuration to process initial risk-screening algorithms globally in under 100 milliseconds with an empirical test accuracy parameter locked at 98.5%. Objective 2—to construct a visually accessible Explainable AI interface for victims—was materialized entirely through the Multi-Agent CrewAI LLM Phase. This system actively empowers simulated analytical personas to transmute rigid mathematical vector inferences into highly readable text summaries, which are dynamically passed through Axios and rendered beautifully upon the React XAIPanel dashboard alongside precise, preventative action steps. Finally, Objective 3—implementing total data integrity and user localization privacy—was achieved flawlessly by stripping the core backend logic of all dependencies connecting to public cloud LLM API providers. By strictly routing the heavily quantized Meta Llama 3B parameter models entirely within strictly firewalled, internal localhost communication ports natively, the system ensures highly confidential voice interaction transcripts never maliciously traverse potentially monitored public tracking domains. 

## 4. Limitation of your project
Despite the major programmatic successes, the rigorous pursuit of strict data privacy inevitably mandated an extremely intensive overhead toll placed upon local CPU and GPU processing compute dependencies. Executing the incredibly intricate Multi-Agent sequential reasoning pipeline completely offline requires deploying the project upon capable local processing unit specifications—expressly incorporating high core-count Desktop CPUs or dedicated neural processing GPUs. This creates massive runtime generation bottlenecks when attempting to execute the application locally upon aging infrastructure or slimline laptop devices, drastically compounding the inference latency during Phase 3 generative generation pipelines. Additionally, the foundational mathematical logic undergirding the entire application remains fundamentally constrained exclusively towards analyzing Anglo-centric English vocabulary matrices. Specifically, the entirety of the TF-IDF feature vocabulary dictionaries, the internal regression anchoring coefficients, and the pre-computed dense vector data of the embedding models operate optimally and exclusively upon English dialect syntax. This effectively restricts the application from neutralizing and parsing foreign, region-specific phishing jargon structures or local-dialect conversational attacks, as localized variations fail to match the english-based historical ChromaDB anchors.

Moving forward, iterative architectural updates and developmental continuations of this project must consider writing and implementing an autonomous, opt-in hybrid-cloud hardware toggle switch inside the user settings panel. By securely engineering federated API pipelines directing network traffic toward highly encrypted, enterprise-grade external graphical processing endpoints—such as Microsoft Azure Cloud nodes or Groq LPU hardware—users severely constrained by inadequate local motherboard configurations could willfully choose to bypass local runtime processing bottlenecks, facilitating instantaneous multi-agent execution computation hosted entirely off-server.
