# CHAPTER 4: SYSTEM DEVELOPMENT

## 4.1 Introduction

This chapter comprehensively outlines the system development phase for the official project titled **VISHING DETECTION USING MACHINE LEARNING AND NLP** (hereinafter referred to by its operational system name, **ShieldGuard**). The primary objective of this chapter is to meticulously document the complex transition from the initial, academically focused FYP1 prototype into a fully functional, highly secure, production-ready system developed during FYP2. The entire development lifecycle strictly followed the Hybrid Rapid Application Development (RAD) methodology. This highly agile, iterative approach was deliberately chosen because it allowed for the continuous, rapid testing and improvement of both the underlying machine learning models and the graphical user interface, perfectly aligning with modern software engineering best practices (Alshathri et al., 2023). 

In the preceding FYP1 phase, the project successfully established a highly accurate mathematical foundation by creating a localized Streamlit-based prototype. This early version evaluated several machine learning approaches, including Support Vector Machine (SVM), Logistic Regression, Random Forest, and a lightweight neural network, to analyze dense text transcripts generated by Automatic Speech Recognition (ASR). After comparison, SVM was selected as the final production classifier because it delivered the strongest practical balance of accuracy, speed, probability calibration, and explainability. This initial prototype was highly successful in a controlled laboratory setting, achieving an impressive mathematical accuracy of 98.51% (Roy & Shin, 2023). However, while the underlying mathematical calculations were highly accurate, the Streamlit framework itself was inherently too simplistic and restrictive for a real-world, commercial cybersecurity application. It struggled severely with managing complex application states, lacked the ability to securely handle multiple concurrent users at once, and could not easily integrate with complex external databases (Nguyen & Le, 2024).

To decisively overcome these architectural limitations and elevate the system into a truly robust, "WOW"-factor cybersecurity platform, FYP2 introduces three major, transformative architectural improvements. First, the system underwent a **Full-Stack Decentralization**, migrating to a split architecture that uses React.js for the visual frontend and FastAPI for the backend, allowing the system to handle heavy user traffic without crashing (Wang et al., 2024). Second, the project implemented **Hybrid Intelligence (ML + LLM)**, combining the rapid mathematical probability of traditional Machine Learning with the deep, human-like psychological reasoning of a Large Language Model (Gupta & Kumar, 2025). Finally, **Enterprise-Grade Security** was integrated into every layer, utilizing JSON Web Tokens (JWT) and PostgreSQL database logging to ensure total user data confidentiality and protection against malicious hackers (Sharma et al., 2023). These upgrades transform the project from a simple proof-of-concept into a layered, real-time defense platform heavily supported by recent cybersecurity research (Chen et al., 2024).

## 4.2 Preparation and Environment Setup

Before a single line of production code could be written, the development environment had to be rigorously and carefully prepared. Building and running deep learning neural networks, localized Large Language Models (LLMs), and a high-speed web server simultaneously on the exact same physical machine requires immense computational power and careful resource management. This step was incredibly vital because, unlike typical web applications that rely on cloud servers, this project focuses heavily on user privacy and data security (Zhang et al., 2023). 

By forcing all AI processing to occur locally on the machine, the system ensures that highly sensitive, potentially confidential audio recordings of phone calls never leave the user's device to be processed by third-party companies. While this "local-first" approach provides the ultimate level of data confidentiality, it places a massive burden on the hardware. If the hardware is insufficient, the system will experience catastrophic memory paging, causing the entire operating system to freeze or crash during voice transcription or LLM text generation (Lee & Kim, 2024).

### 4.2.1 Hardware Specifications

The hardware environment was deliberately chosen to perfectly balance the urgent need for substantial Random Access Memory (RAM) and the computational necessity of Graphics Processing Unit (GPU) acceleration. The RAM acts as the system's "short-term memory", holding the massive AI models ready for instant use, while the GPU acts as an incredibly fast calculator for processing complex neural networks.

If the system had relied purely on a standard Central Processing Unit (CPU) for all tasks, the audio transcription process could take several minutes per file, utterly destroying the goal of near-real-time vishing detection. Therefore, a hybrid hardware approach was required, leveraging both high-core CPUs for web routing and dedicated GPUs for AI math (Patel et al., 2023).

## 4.3 System Architecture Overview

The ShieldGuard system completely abandons simple, fragile designs in favor of a modern, enterprise-grade "Three-Tier Architecture". To understand this concept, one can visualize a high-end restaurant: the React Frontend acts as the beautifully decorated dining area where users interact; the FastAPI Backend acts as the highly efficient waitstaff that securely takes and manages orders; and the Machine Learning models act as the intense, specialized kitchen preparing the final results (Nguyen & Le, 2024). This strict separation ensures that if the frontend interface crashes, the backend AI models remain completely secure and unaffected.

The most robust, "WOW"-factor feature of this entire architecture is its **Layered Cascade Design**. Instead of relying lazily on just one single AI model to do all the work, the system forces the incoming data through multiple, distinct security checkpoints. First, the traditional Machine Learning (ML) model mathematically calculates the exact risk percentage based on vocabulary. Only if the mathematical risk is elevated is the data then passed to the Large Language Model (LLM) to extract and explain the psychological threat to the user in plain English. 

This specific hybrid arrangement is critically important because Large Language Models are infamous for a flaw called "hallucination"—where they confidently make up false information (Huang et al., 2023). By placing the deterministic, mathematical ML model as the ultimate authority, the system effectively handcuffs the LLM. It guarantees that a hallucinating LLM can never accidentally declare a mathematically proven, highly dangerous vishing call as "safe", ensuring the user is always protected by hard mathematics first and generative text second.

## 4.4 Development Phase 1: Full-Stack Migration and Security

The first major phase of development focused exclusively on building the robust web infrastructure and locking it down with uncompromising, enterprise-grade security. Moving completely away from the basic, vulnerable FYP1 prototype, FastAPI was chosen as the system's backbone. This is because FastAPI can handle heavy, time-consuming AI processing in the background asynchronously, ensuring that the user's web browser never freezes while waiting for a result (Wang et al., 2024).

However, building a powerful API is meaningless if it can be easily hacked. Therefore, to ensure the system is entirely safe for public use, several advanced cybersecurity defenses were coded directly into the backend architecture from day one. These defenses were designed specifically to uphold the CIA Triad—Confidentiality, Integrity, and Availability—which is the absolute gold standard in global cybersecurity frameworks (Sharma et al., 2023).

### 4.4.1 Implementing Enterprise-Grade Security Controls

The system actively defends itself against automated botnets, credential stuffing, and malicious code injections. Rather than using traditional, easily stolen cookies for user login sessions, the system uses stateless cryptography.

Furthermore, the system assumes that all user input is inherently dangerous. If a hacker attempts to paste an executable virus script into the transcript text box (a technique known as Cross-Site Scripting or Prompt Injection), the system's input sanitization algorithms will instantly neutralize the threat before the AI even reads it.

**Table 12: Implemented Cybersecurity Defenses and CIA Triad Alignment**

| Implemented Defense Mechanism | Technical Execution Details | CIA Triad Alignment & Why It Matters |
| :--- | :--- | :--- |
| **JSON Web Token (JWT) Authentication** | The server issues cryptographically signed digital tokens that mathematically expire after exactly 24 hours. | **Confidentiality:** Acts as a secure, unforgeable digital ID card. It completely prevents hackers from hijacking an active user's session over the network. |
| **Algorithmic Brute-Force Lockout** | The database automatically locks an account out if it detects 5 consecutive failed password attempts within a 15-minute window. | **Integrity:** Stops automated botnets from using "dictionary attacks" to guess a user's password thousands of times a second. |
| **API Rate Limiting** | The gateway server actively tracks and restricts users to a maximum of 30 heavy AI analyses per hour. | **Availability:** Prevents a single malicious user from spamming the system and crashing the expensive AI servers, ensuring the system remains online for everyone else. |
| **Aggressive Input Sanitization** | Uses Regular Expressions (RegEx) to forcefully strip out all hidden code, HTML tags, and markdown from text inputs. | **Integrity:** Prevents Cross-Site Scripting (XSS), stopping hackers from sneaking malicious scripts into the text box to infect the server or trick the LLM. |
| **Cryptographic Password Hashing** | Uses the computationally heavy `bcrypt` algorithm with 12 mathematical rounds to scramble user passwords. | **Confidentiality:** Ensures that even if the entire Supabase database is somehow stolen by hackers, they absolutely cannot read or reverse-engineer the users' real passwords. |

## 4.5 Development Phase 2: ML Pipeline Hardening

The second phase of development pivoted entirely away from the web interface to focus exclusively on the complex mathematics and Natural Language Processing (NLP) of the AI. A major, highly destructive challenge in machine learning is a phenomenon known as "data drift" or "pipeline mismatch." This occurs when the text analyzed by the live production system is cleaned slightly differently than the text originally used to train the system in the laboratory. 

If these preprocessing pipelines do not match byte-for-byte, the AI becomes hopelessly confused. In an early test, failing to properly standardize punctuation caused the system to panic, resulting in a disastrous 97.8% false positive rate on perfectly safe calls (Ma et al., 2025). This proved that raw mathematical accuracy is useless without perfect data hygiene.

### 4.5.1 Advanced Natural Language Processing (NLP)

To permanently fix this vulnerability, the system was upgraded to use strict, unyielding NLP techniques to mathematically clean and standardize every single sentence before the AI is allowed to read it (Hassan et al., 2024). 

This cleaning process is rigorous. First, it uses **Lowercasing and Punctuation Removal** to convert all text to a uniform baseline, removing commas and periods so the AI doesn't incorrectly think the words "Bank" and "bank!" are two entirely different concepts. Second, it utilizes advanced **Lemmatization** via the NLTK library. Lemmatization acts like an automated dictionary, reducing complex words down to their absolute base form (e.g., stripping "transferring" or "transferred" down to just "transfer"). This drastically shrinks the vocabulary size and makes it incredibly easy for the AI to recognize the mathematical patterns of a scam.

### 4.5.2 Asymmetric Thresholds and Explainable AI (XAI)

Most basic, amateur AI systems lazily flag anything over a 50% probability as dangerous. However, in real-world cybersecurity, false alarms are incredibly destructive because they annoy users, eventually causing them to ignore the system entirely (alert fatigue). To intelligently prevent this, ShieldGuard implements a highly advanced concept called an **"Asymmetric Threshold"** (Provost & Fawcett, 2022). The system is programmed to only officially declare a call as a dangerous scam if the SVM model is more than 80% confident, heavily prioritizing user peace-of-mind over hyper-sensitivity.

Furthermore, to ensure the system isn't just an untrustworthy "black box" (where the AI makes a decision but cannot explain why), robust Explainable AI (XAI) features were engineered into the dashboard. The system dynamically reads the SVM's mathematical weights and highlights the specific dangerous keywords in bright red (such as "OTP", "Urgent", or "Arrest") directly on the screen. This allows the user to easily and visually understand exactly why the system flagged the text, building immense trust between the human and the machine (Gupta & Kumar, 2025).

## 4.6 Development Phase 3: Hybrid Intelligence (RAG & LLM)

The third, final, and undeniably most impressive phase of development was integrating the localized Large Language Model (LLM). While the traditional ML model built in Phase 2 is magnificent at high-speed mathematics, it completely lacks the ability to understand human psychology or context. 

The LLM acts as an incredibly smart, tireless human advisor. Its sole job is to read the transcript and explain the specific psychological manipulation of the scam to the user in plain, easy-to-understand English (Chen et al., 2024). However, LLMs are dangerous if left unchecked, so they must be heavily guided and restricted to ensure they are helpful rather than harmful.

### 4.6.1 Retrieval-Augmented Generation (RAG)

Before the LLM is even allowed to generate an explanation, the system forcefully utilizes a cutting-edge technique called Retrieval-Augmented Generation (RAG). RAG can be easily understood by comparing it to giving a student an "open-book exam" rather than forcing them to rely on their potentially flawed memory (Gao et al., 2023). 

Instead of letting the LLM guess, the system takes the user's transcript, mathematically searches the ChromaDB vector database for the two most highly similar historical scams on record, and gives those exact historical examples to the LLM. This "grounds" the AI, physically forcing it to base its advice on verified, factual, real-world data rather than hallucinating or making things up.

### 4.6.2 The Divergence Detector (Preventing Hallucinations)

As previously mentioned, Large Language Models suffer from "hallucination"—they will occasionally and very confidently give completely wrong answers (Huang et al., 2023). In a cybersecurity application, an AI telling a user that a dangerous scam is safe could lead to devastating financial loss.

To protect the user at all costs, a highly complex **"Divergence Detector"** algorithm was coded into the system. This algorithm constantly watches both AIs. If the highly accurate mathematical ML model states the call is a dangerous scam (e.g., 95% risk), but the LLM hallucinates and generates a text response saying "This call seems perfectly safe," the system instantly detects the massive disagreement. It aggressively overrides the LLM, deletes the false text, and alerts the user that the call is highly suspicious, proving that in ShieldGuard, cold hard math always overrules generative text.

## 4.7 Project Structure and Version Control

To ensure the project is highly professional, easily maintainable, and ready for future upgrades by other developers, the entire codebase was organized logically into strict, isolated directories. This is a massive improvement over amateur projects that dump all code into a single, confusing folder.

**Table 13: Enterprise Project Directory Architecture**

| System Directory | Deep Purpose and Organizational Rationale |
| :--- | :--- |
| `frontend/src/` | Contains all the visual JavaScript code (React.js) that creates the user interface. It is kept completely separate so frontend developers can work without breaking the backend AI. |
| `backend/` | Contains the FastAPI routing logic that safely connects the user's web browser to the heavy Python AI models running on the server. |
| `models/` | The highly secure storage folder for the trained, serialized ML models (like the SVM). By isolating them here, they can be easily loaded directly into the server's RAM upon startup. |
| `data/` | Stores the ChromaDB vector database, which contains thousands of verified historical scams used to provide the "open-book exam" data for the RAG system. |

Furthermore, for strict version control, **Git** was heavily utilized throughout the entire year. Git acted as an incredibly powerful "save state" for the thousands of lines of code written. Whenever a major new feature was developed (such as adding the LLM), the development occurred on a separate "branch". If the new code accidentally broke the system, Git allowed the development to easily and instantly roll back the entire project to the previously working version, ensuring the system was never permanently damaged during development (Alshathri et al., 2023).

## 4.8 Summary

This chapter has comprehensively and transparently explained exactly how the VISHING DETECTION USING MACHINE LEARNING AND NLP system was engineered from the ground up. The development was structured into three highly logical, cascading phases that transformed a basic prototype into a true technological marvel.

Phase 1 completely secured the system's foundation by implementing a lightning-fast FastAPI server, locking it down with JWT cryptography, brute-force protections, and secure database logging to ensure it met strict global cybersecurity standards. Phase 2 hardened the NLP pipeline, using precise text cleaning algorithms and brilliant asymmetric thresholds to prevent annoying false alarms and establish absolute mathematical trust. 

Finally, Phase 3 introduced the true "WOW" factor of the project: a pioneering Hybrid RAG + LLM engine. This engine brilliantly explains highly complex scams to the user in plain English, while utilizing a rigid Divergence Detector to mathematically prevent the inherent dangers of AI hallucinations. Together, these meticulous development phases have resulted in a highly robust, incredibly accurate, and exceptionally user-friendly defense platform capable of protecting everyday users from devastating social engineering attacks.
